# 案1〜3 挙動追加に伴う将来問題 予測レポート

**作成日**: 2026-07-04
**対象**: 案1(Servo Realism)、案2(Mode Switching)、案3(Day/Night Consistency)
**位置づけ**: Level1/2実装で実際に発生した単位不一致バグ・定常偏差の教訓を踏まえ、今後の挙動追加で「発生しうる問題」と「発生しない問題」を事前に整理する。実装前の予測であり、実装時に答え合わせをする前提の資料。

---

## 前提：Level1/2で得た2つの教訓

今後の予測は、実際に踏んだ以下2つの地雷を出発点にする。

**教訓1（単位不一致）**: `compiler angle="degree"`はjointの`range`/`ref`には効くが、actuatorの`ctrlrange`には効かない。この結果、目標90度に対し実際171度・195度に収束する不具合が発生した。原因は`kp*(ctrl-qpos)`の力計算で、ctrl側だけ度、qpos側がradianという単位混在だった。

**教訓2（定常偏差）**: position actuator(P制御のみ、I項なし)は、片持ちカメラの重力トルクを完全には打ち消せず、TILT軸に約0.15〜7度の定常偏差が生じる。これは実機`AxisPdController`がI項を持たない設計と整合する物理的必然であり、バグではなく仕様である。

この2つは性質が異なる。教訓1は**実装ミスで発生する問題**（正せば消える）、教訓2は**制御則の構造そのものに起因する問題**（構造を変えない限り残り続ける）。以下の予測でもこの区別を用いる。

---

## 案1: Servo Realism Benchmark（PID化・遅延・backlash・deadband・quantization）

### 発生する問題

**1-A. デッドゾーン判定とMuJoCoのctrl保持機構の不整合**

実機コード(`inference.cpp`)は次の判定でデッドゾーンを実装している。

```cpp
const bool yaw_in_dead = std::abs(yaw_axis_.prev_err_deg) < kYawDeadZoneDeg;
if (!yaw_in_dead || !pitch_in_dead) {
    publishState(...);   // 指令を送る
}
// dead zone内なら publishState を呼ばない = 前回の指令値が保持される
```

これをMuJoCo側で素朴に「誤差が閾値未満ならこのステップは`data.ctrl`を更新しない」と実装した場合、**MuJoCoの`ctrl`はPython側が明示的に上書きしない限り前の値を保持し続ける**ため、一見同じ挙動に見える。しかし実際に検証すると、`env.reset()`直後の`ctrl`は`[1.57079633, 1.57079633]`(=90.000度)なのに対し、収束後の`qpos`は`89.99998128`度と**厳密には一致しない**。デッドゾーン判定を「目標角と現在角の差」で行うか「前回誤差prev_err_deg」で行うかによって、判定に使う値がわずかに異なり、実機のロジックをそのまま移植したつもりでも判定タイミングがずれる可能性が高い。これは**実装時に必ず突き合わせ検証が必要**な箇所であり、教訓1と同種の「地道な単位・タイミング不一致」が起きやすい。

**1-B. dt正規化の固定タイムステップ問題**

実機の`dt`はROSコールバックの実行間隔（シリアル遅延・YOLO推論時間・OS スケジューリングにより変動）だが、MuJoCoの`timestep`は`0.002`固定である。もし「mj_stepをNステップごとに制御更新」という構造で組んだ場合、**そのdtは常に同じ値になり、実機のdt揺らぎに起因するD項スパイクが原理的に再現できない**。これは実装ミスではなく、**離散時間シミュレーションの本質的な制約**であり、案1が目指す「D項スキップが必要になるほどのdt不定性」を意図的にノイズとして注入しない限り発生しない(=再現できない)。対策として、dtに人工的なジッタを加える実装が必要になる。

**1-C. backlashのXML表現不能問題**

Q2で既に指摘した通り、backlash(遊び)は単一のhinge jointでは表現できない。2つのjointをバネで疎結合させる構造が必要で、これは**モデル構造(XML)自体の変更**を要する。「XMLは終始不変」の原則が破られる最初のケースになる。

**1-D. quantization実装時の丸め方向の不一致**

Q1レポートで発見した通り、実機は`static_cast<int>(msg->x_angle)`で**単純な切り捨て**を行っている（四捨五入ではない）。MuJoCo側で`round()`を使うと実機と異なる丸め方向になり、境界値（例: 90.5度）で1度のズレが生じる。地味だが、教訓1と同種の「気づきにくい数値不一致」である。

### 発生しない問題

**1-E. サーボ飽和・トルク不足**

D1/B1で確認済みの通り、TILT/PANとも安全率30〜59倍と極めて余裕があるため、PID化やdeadband追加で指令パターンが変わっても、**トルク不足や飽和による異常挙動は起きない**。案1のどの挙動を足しても、この点は心配不要。

**1-F. 積分項(I)によるwind-up問題は起きにくい**

PXは可動範囲が±60度程度の小さい系で、目標到達までの時間も短い(b3実測で1秒未満)。一般的なPID制御で問題になる「積分項の蓄積し過ぎ(wind-up)」は、可動範囲が広くsaturationが長時間続く系で顕著になる現象であり、PXの規模では**発生確率は低い**。ただし全くのゼロではないため、I項を足す場合はclampを入れる方が安全ではある。

## 案2: Mode Switching（watch/detect/search FSM）

### 発生する問題

**2-A. mode遷移とアクチュエータ状態の不整合**

実機の`enterDetectTrackMode()`は次を実行する。

```cpp
void OnnxInferenceNode::enterDetectTrackMode() {
  mode_ = Mode::DETECT_TRACK;
  no_det_frames_ = 0;
  first_detect_frame_ = true;
  yaw_axis_.reset(0.0);
  pitch_axis_.reset(0.0);
  publishWakeup(true);
}
```

`yaw_axis_.reset(0.0)`で`prev_err_deg`が0にリセットされる。MuJoCo環境クラス(`PxPanTiltEnv`)にmode遷移を実装する際、**リセットするタイミングを1ステップでも誤ると、次のD項計算で実際の誤差との差分が異常値になる**。これは1-Bのdt=0処理と連動しており、「mode遷移の瞬間」は複数のリセット処理が同時に必要になる箇所で、実装順序を誤りやすい。

**2-B. watch mode中のfps間引きとmj_stepの粒度不一致**

実機のWATCHモードは`watch_fps=2.0`で意図的にフレームを間引く。

```cpp
const double fps = (watch_fps_ > 0.0 ? watch_fps_ : 2.0);
const auto period = std::chrono::duration<double>(1.0 / fps);
```

これは「500msに1回だけ画像処理をする」という**実時間ベース**の間引きである。MuJoCoの`mj_step`は物理タイムステップ(2ms)ベースで動くため、「2Hzの観測」を「250 mj_stepごとに1回」のように**ステップ数で近似する**実装になる。この変換自体は単純だが、案2で検出ノイズやドロップアウトを追加した際、**「観測がない間、サーボ指令は何をすべきか」（前回値保持か、指令送信自体を止めるか）の仕様が実機コード上で明示されていない**ため、実装者側の解釈で挙動が分かれる。これは新しい問題というより、**実機コード自体の仕様の曖昧さがシミュレーション実装時に表面化する**ケースである。

**2-C. search modeは実機に存在しないため「正解」がない**

Q1レポートに記載の通り、実機には`search mode`が実装されていない（見失うとWATCHに戻るのみ）。案2で新たにsearch modeを設計する場合、**比較対象となる実機の挙動が存在しない**ため、b3のような「実機ログとの定量比較」ができない。これは技術的な不具合ではなく、**検証方法そのものを新規設計しなければならない**という構造的な制約である。

### 発生しない問題

**2-D. PIRの二重デバウンス起因の問題**

Q1レポートで確認した`watch_node`(200msデバウンス)と`px3_servo_node`(エッジ検知)の二重デバウンス構造は、**PIRセンサーというハードウェア入力に固有の問題**であり、MuJoCo上で「仮想ターゲットの出現」をシミュレートする限りにおいては、この二重構造をそのまま模す必要がない。PIRの代わりにMuJoCo側で単純なブール値イベントを発生させれば足りるため、**この複雑さはシミュレーションには持ち込まれず、問題として発生しない**。

**2-E. Arduino起動待機(4秒)、シリアルタイムアウト(1秒)関連の問題**

これらは物理シリアル通信に起因する待機時間であり、MuJoCo環境には対応する概念自体が存在しない。案2のFSM設計にこれらを模す必要はなく、**発生しようがない**。

## 案3: Day/Night Consistency（Pattern-B/D比較）

### 発生する問題

**3-A. 質量差に起因するゲイン不適合（既に部分的に発覚済み）**

b1(Pattern-D, TILT安全率59倍)とb3(Pattern-B, 同35倍)の比較から、**同一のPDゲイン(kp=12, damping=0.5等)を両モデルに適用した場合、Pattern-Bの方がオーバーシュート・定常偏差が大きくなる**ことは既に部分的に確認されている。案3を本格的に実施すると、この差が「無視できる範囲か、再チューニングが必須か」の**定量的な閾値**を出す必要が生じる。閾値の基準（例: 定常偏差5%以内なら許容、等）は実機コード上に存在しないため、**新たに評価基準を定義しなければならない**。これは2-Cと同種の「比較対象が実機に存在しない」問題である。

**3-B. 統合XMLと旧pattern_d.xmlの構造不一致**

現行の`pattern_b_integrated.xml`はQ3で確定した「分割XML + include」方式だが、以前作成した`pattern_d.xml`は単一ファイル完結型である。両者を同一スクリプトから比較する場合、**body名・joint名の命名規則が異なる**(`yaw`/`pitch` vs `pan`/`tilt`)ため、比較スクリプトが両モデルの差異を吸収する抽象化層を持たない限り、**モデルごとに個別分岐が必要になる**。これは教訓1のような単位不一致ではないが、**保守性の低下という別種の問題**として発生する。

### 発生しない問題

**3-C. カメラ質量そのものの起因する物理破綻**

D1/B1で確認済みの通り、Pattern-D(可動部合計約220g)・Pattern-B(約230g)とも、MG996Rのトルク余裕は数十倍あるため、**どちらのモデルでもトルク不足による物理的な破綻(サーボが目標に到達できない等)は起きない**。案3の比較はあくまで「ゲインの相対的な適合度」の問題であり、絶対的な動作可否の問題ではない。

**3-D. 上下対称配置のトルク相殺効果の消失**

Pattern-D特有の「IR上/RGB下による重力モーメント相殺」(D1で安全率59倍と確認)は、Pattern-Bには存在しない設計思想である。案3で両者を比較する際、**この違いは「バグ」ではなく「設計の違いが定量的に見える」という案3の本来の目的そのもの**であるため、問題としてではなく、むしろ**期待される検証結果**として扱ってよい。

## 全体まとめ表

| 案 | 発生する問題(要対処) | 発生しない問題(心配不要) |
|---|---|---|
| 案1 | デッドゾーン判定のタイミング不一致、dt固定によるジッタ再現不可、backlashのXML表現不能、丸め方向の不一致 | トルク不足・飽和、積分wind-up(規模的に起きにくい) |
| 案2 | mode遷移時のリセット順序、fps間引きの実時間→ステップ数変換の仕様曖昧性、search modeの比較対象不在 | PIR二重デバウンスの複雑さ、シリアル通信起因の待機時間 |
| 案3 | ゲイン不適合の評価基準が実機に存在しない、XML命名規則の不一致による保守性低下 | トルク不足による物理破綻、対称配置効果の消失(これは期待通りの結果) |

## 教訓1型(実装ミスで発生・正せば消える)とみなせるもの
- 1-A(デッドゾーン判定)、1-D(丸め方向)、3-B(命名規則不一致)

## 教訓2型(構造起因・原理的に残り続ける、または新規設計が要る)
- 1-B(dt固定)、1-C(backlash)、2-A(mode遷移リセット)、2-C(search mode比較対象不在)、3-A(評価基準不在)

後者は「直す」対象ではなく「どう設計するか」を新たに決める対象である、という認識で進めるのが良い。
